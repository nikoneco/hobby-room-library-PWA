# 趣味部屋図書館 改修リスク診断書 v1

作成日: 2026-05-14  
対象: ローカルフォルダ内の Google Apps Script / HTML / CSS / JavaScript 一式  
方針: 現行コードを読み取った結果のみで整理。コード変更・整形・削除・移動は未実施。

## 1. 診断サマリー

現行コードは、検索機能・シリーズ判定・あらすじ取得・画像補完・本棚ビュー・スプレッドシート運用補助が同居しているため、単純な関数整理や列定義変更でも広い範囲に影響が出やすい構造です。

特に注意が必要な領域は以下です。

- `Webアプリ.gs.txt` 内に同名関数 `buildImageUrlFromIsbn_` が二重定義されている
- `CONFIG.COL` はシート上の 1 始まり絶対列、`CONFIG.IDX` は `I:AB` 読み込み後の 0 始まり配列 index であり、混同すると列ズレが起きる
- Webアプリのキャッシュは `目録` 編集時には破棄されるが、`genre_master` / `series_master` / `データ` 側の変更では自動破棄されない可能性がある
- `onEdit` はキャッシュ破棄、画像補完、あらすじ状態管理、シリーズキー更新、検索モード操作、ソート、フィルター初期化をまとめて扱うため副作用範囲が広い
- 本棚ビューは HTML / CSS / JS の id・class・グローバル関数に強く依存している

## 2. リスク一覧

### リスク 1: 同一 GAS ファイル内の `buildImageUrlFromIsbn_` 二重定義

- リスク名: `buildImageUrlFromIsbn_` の二重定義
- 該当ファイル: `Webアプリ.gs.txt`
- 該当関数: `buildImageUrlFromIsbn_`, `buildHanmotoImageUrlFromIsbn_`, `mapRowsToBooks_`
- 何が危険か: 同じ名前の関数が同一ファイル内で 2 回定義されている。JavaScript / Apps Script では後から定義された関数が有効になるため、前半の実装は実行上上書きされる。
- 実害が出る条件: 画像 URL 生成処理を改修する際、前半の `buildImageUrlFromIsbn_` を修正しても実際には後半の定義が使われる。逆に削除・統合時に意図せず挙動を変える可能性がある。
- 修正優先度: 高
- 修正するなら安全な方針: まず現行で実際に呼ばれている後半の定義を基準にする。前半と後半の戻り値が同等か確認し、テスト用ログまたは小さな検証関数で ISBN あり・なし・不正値を比較してから、片方を削除または名前変更する。
- 今は触らない方がよい場合はその理由: 画像表示のフォールバック処理に関わるため、初回の整理段階では削除しない。見た目上は重複でも、過去の改修履歴や意図が残っている可能性がある。

### リスク 2: サーバー側とクライアント側に同名の正規化関数がある

- リスク名: `normalizeKana` / `hiraToKana` の同名別実装
- 該当ファイル: `config.gs.txt`, `script.js.html`
- 該当関数: `normalizeKana`, `hiraToKana`, `normalizeText`, `matchesPreviewConditions`, `countPreviewMatchesAuthoritative`
- 何が危険か: サーバー側とクライアント側で同じ名前の関数が存在するが、実行環境も細部の処理も異なる。サーバー側は記号除去が広く、クライアント側はプレビュー用の簡易正規化に近い。
- 実害が出る条件: クライアント側プレビュー件数と、サーバー側の本検索件数が一致しないことがある。正規化処理を片側だけ変更すると、検索結果・確認中件数・実検索件数に差が出る。
- 修正優先度: 中
- 修正するなら安全な方針: 先に「プレビューは近似、最終検索はサーバー authoritative」という役割を明文化する。完全一致を目指す場合は、検索条件ごとのサンプルを作って件数差を検証してから合わせる。
- 今は触らない方がよい場合はその理由: 現行コードには `countPreviewMatchesAuthoritative` があり、プレビュー差を補正する設計が入っている。正規化を安易に統一すると、既存の体感速度や検索挙動が変わる可能性がある。

### リスク 5: `CONFIG.COL` と `CONFIG.IDX` の混同

- リスク名: 絶対列番号と配列 index の混同
- 該当ファイル: `config.gs.txt`, `Webアプリ.gs.txt`, `コード.gs.txt`, `あらすじ取得_kobo.gs.txt`, `あらすじ取得_Main.gs.txt`, `Web画像取得.gs.txt`
- 該当関数: `loadMainBookData_`, `mapRowsToBooks_`, `getLastDataRow`, `onEdit`, `markSynopsisManualOnEdit_`, `batchFetchSynopsisRawByLimit_`, `batchFillFallbackImageUrlsByLimit_`
- 何が危険か: `CONFIG.COL` はシート上の 1 始まり列番号、`CONFIG.IDX` は `I:AB` 範囲を読み込んだ後の 0 始まり配列 index。用途が異なるため、取り違えると別列を読んだり書いたりする。
- 実害が出る条件: タイトルのつもりで別列を読む、画像 URL のつもりで ISBN を読む、あらすじ RAW や取得元を誤列に書く、シリーズキーやジャンルがずれる。
- 修正優先度: 高
- 修正するなら安全な方針: シート操作では `CONFIG.COL`、`I:AB` の row 配列操作では `CONFIG.IDX` というルールをコメントではなく仕様書・チェックリストに固定する。改修前に対象関数内の `getRange` 開始列と row index の対応を必ず確認する。
- 今は触らない方がよい場合はその理由: 現在は列定義に依存した処理が広範囲にある。列追加・列移動・読み込み範囲変更は、検索・画像・あらすじ・ジャンル・本棚ビューすべてに影響する。

### リスク 6: `CONFIG.COL.AUTHOR` を出版社取得の index として使っている箇所がある

- リスク名: 出版社列参照の意味づけが不明瞭
- 該当ファイル: `あらすじ取得_kobo.gs.txt`
- 該当関数: `scoreRakutenKoboItemMatch_`
- 何が危険か: コメント上では `CONFIG.COL.PUBLISHER` が未定義のため `AUTHOR+1` 相当を使う意図だが、実コードは `row[CONFIG.COL.AUTHOR]` を出版社として読んでいる。これは A 列開始の全行配列では L 列に当たり、結果として出版社列を指している可能性があるが、読み手には非常に誤解されやすい。
- 実害が出る条件: `CONFIG.COL.AUTHOR` を「K列著者」と理解して修正すると、Kobo 候補スコアリングで出版社照合が壊れる。`getRange` の読み込み開始列が変わっても壊れやすい。
- 修正優先度: 中
- 修正するなら安全な方針: 直接修正前に、対象関数へ渡される `row` が A 列開始か I 列開始かを確認する。安全に直すなら `CONFIG.COL.PUBLISHER` を定義し、A 列開始配列用の index 計算を明示する。
- 今は触らない方がよい場合はその理由: Kobo 候補選定のスコアに関わるため、動作確認なしで変えるとあらすじ取得品質が下がる可能性がある。

### リスク 7: マスター系シート変更時のキャッシュ破棄漏れ

- リスク名: `genre_master` / `series_master` / `データ` 変更後の Web 表示 stale
- 該当ファイル: `コード.gs.txt`, `Webアプリ.gs.txt`, `config.gs.txt`
- 該当関数: `onEdit`, `clearLibrarySearchCache`, `getLibraryDataset_`, `loadGenreMasterMap_`, `loadSeriesMasterMap_`, `getAdvancedSearchOptions`
- 何が危険か: Web アプリ用データセットはキャッシュされる。`onEdit` は `目録` 編集時にキャッシュを破棄するが、`genre_master` / `series_master` / `データ` 側の変更で自動破棄される処理は確認できない。
- 実害が出る条件: ジャンルマスターやシリーズマスター、検索条件用のデータを更新しても、Web アプリ側に古いジャンル・シリーズ表示・候補が残る。TTL 経過または手動キャッシュ削除まで反映されない。
- 修正優先度: 高
- 修正するなら安全な方針: まずキャッシュに入るデータの範囲を確定する。次に `onEdit` で対象シートと対象列だけを見て `clearLibrarySearchCache` を呼ぶか、手動更新メニュー・明示ボタン方式にする。いきなり全シート編集で破棄すると編集体験が重くなる可能性がある。
- 今は触らない方がよい場合はその理由: キャッシュ破棄頻度を上げると Web アプリ初回表示が重くなる可能性がある。反映速度と速度のバランス設計が必要。

### リスク 8: V列「あらすじ」と Y列「あらすじ_RAW」の優先関係

- 今は触らない方がよい場合はその理由: あらすじ表示の仕様変更になる。現行表示が V 優先で成立している可能性もあるため、データ確認後に扱う。
